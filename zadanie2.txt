ЗАДАНИЕ 2. Проектирование архитектуры CSRF-защиты

1. Где хранить токен на сервере. Почему не в БД, а в сессии.

Токен хранится в сессии пользователя: $_SESSION['csrf_token'].

Почему не в базе данных:
- Токен нужен только на время сессии пользователя. Сессия сама удаляется.
- База данных для временных данных — лишняя нагрузка (каждый POST-запрос будет делать запрос к БД).
- В сессии проверка происходит мгновенно, без дополнительных обращений.
- Не нужно писать код для очистки старых токенов.

2. Схема жизненного цикла токена (сплошная цепочка)

1. ГЕНЕРАЦИЯ => 2. СОХРАНЕНИЕ НА СЕРВЕРЕ => 3. ПЕРЕДАЧА КЛИЕНТУ => 4. ХРАНЕНИЕ В ФОРМЕ => 5. ОТПРАВКА С ЗАПРОСОМ => 6. ПРОВЕРКА НА СЕРВЕРЕ => 7. ПРИ НЕСОВПАДЕНИИ

ОПИСАНИЕ КАЖДОГО ЭТАПА:

1. ГЕНЕРАЦИЯ: Сервер вызывает функцию generate(), которая создаёт случайную строку длиной 64 символа.

2. СОХРАНЕНИЕ НА СЕРВЕРЕ: Токен сохраняется в сессию в переменную $_SESSION['csrf_token'].

3. ПЕРЕДАЧА КЛИЕНТУ: Сервер вызывает функцию embed(), которая возвращает HTML скрытого поля с токеном.

4. ХРАНЕНИЕ В ФОРМЕ: Браузер получает HTML и хранит токен в DOM страницы.

5. ОТПРАВКА С ЗАПРОСОМ: Пользователь отправляет форму, браузер добавляет токен в POST-данные.

6. ПРОВЕРКА НА СЕРВЕРЕ: Сервер получает токен из $_POST и вызывает validate($token), сравнивая с токеном из сессии.

7. ПРИ НЕСОВПАДЕНИИ: Если токены не равны, validate() возвращает false, сервер отвечает 403 Forbidden и не выполняет действие.

4. HTTP-ответы при проверке токена

- При успешной проверке (validate вернул true):
  Сервер возвращает код 200 OK и выполняет действие (создаёт, удаляет или обновляет данные).

- При неуспешной проверке (validate вернул false):
  Сервер возвращает код 403 Forbidden и НЕ выполняет действие.

Почему 403:
400 Bad Request означает ошибку в синтаксисе запроса. 
403 Forbidden означает, что сервер понял запрос, но отказывается его выполнять по соображениям безопасности. Неверный CSRF-токен — это именно запрет доступа.
