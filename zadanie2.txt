ЗАДАНИЕ 2. Проектирование архитектуры CSRF-защиты

Вопрос 1. Где хранить токен на сервере. Почему не в базе данных, а в сессии.

Токен будет храниться в сессии пользователя, в переменной $_SESSION['csrf_token'].

Почему не в базе данных:
- Токен нужен только пока пользователь находится на сайте. Как только он закрыл браузер, токен становится не нужен. Сессия живёт ровно столько же, сколько нужно.
- База данных хранит данные долго. Если хранить токены в БД, они будут накапливаться, и их придётся чистить вручную.
- Каждый раз, когда пользователь отправляет форму, серверу пришлось бы делать запрос к базе данных, чтобы проверить токен. Это медленно и создаёт лишнюю нагрузку.
- Сессия уже есть у каждого авторизованного пользователя. Данные сессии хранятся на сервере в файлах или в памяти. Проверка токена из сессии происходит мгновенно, без запросов к БД.

Вопрос 2. Схема жизненного цикла токена

1. ГЕНЕРАЦИЯ
   Пользователь открывает страницу с формой. Сервер создаёт случайную строку из букв и цифр длиной 64 символа.

2. СОХРАНЕНИЕ НА СЕРВЕРЕ
   Сервер сохраняет эту строку в сессию пользователя.

3. ПЕРЕДАЧА КЛИЕНТУ
   Сервер вставляет эту строку в HTML-форму как скрытое поле. Пользователь его не видит, но оно есть в коде страницы.

4. ХРАНЕНИЕ В ФОРМЕ
   Браузер хранит эту строку в памяти страницы. Когда пользователь заполняет форму, токен остаётся внутри неё.

5. ОТПРАВКА С ЗАПРОСОМ
   Пользователь нажимает кнопку "Отправить". Браузер отправляет на сервер все поля формы, включая скрытое поле с токеном.

6. ПРОВЕРКА НА СЕРВЕРЕ
   Сервер получает токен из формы и сравнивает его с токеном, который хранится в сессии.

7. ЧТО ДЕЛАТЬ ПРИ НЕСОВПАДЕНИИ
   Если токены не совпадают или токена нет, сервер НЕ выполняет действие (не создаёт, не удаляет, не обновляет). Вместо этого сервер возвращает ошибку 403 и показывает сообщение "Неверный CSRF токен". Сессию при этом можно не удалять, чтобы пользователь мог попробовать снова.

Вопрос 4. Какие HTTP-ответы должен возвращать сервер

При успешной проверке токена:
Сервер возвращает HTTP-код 200 OK.
Это означает, что всё в порядке, запрос правильный. Сервер выполняет действие: создаёт объявление, удаляет или обновляет профиль. После этого возвращает пользователю сообщение об успехе.

При неуспешной проверке токена (токен неверный или отсутствует):
Сервер возвращает HTTP-код 403 Forbidden.
Это означает "Запрещено". Сервер понял запрос, но отказывается его выполнять по соображениям безопасности. Сервер НЕ меняет никакие данные и возв


ращает сообщение об ошибке.

Почему именно 403, а не 400:
- 400 Bad Request означает, что сервер не понял запрос из-за ошибки в его формате (например, кривые заголовки или неправильный JSON).
- 403 Forbidden означает, что сервер понял запрос, но у отправителя нет прав на это действие или нарушены правила безопасности.
- Неверный CSRF-токен — это именно ситуация "сервер понял запрос, но не доверяет ему". Поэтому правильный код ответа — 403.
